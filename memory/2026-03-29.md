## Session après-midi — Nouveau script Pine v5 générique (HMA Cross + Labouchere)

### Contexte

- Fichiers : `trading/hma_cross_strategy.pine` + `trading/hma_cross_indicator.pine`

### Bugs Pine Script v5 rencontrés et corrigés (chronologie)
1. **Zéro trades** → limite orders jamais remplies → corrigé : market orders
2. **Exécution à la clôture** → voulu : `process_orders_on_close=true`
3. **JP veut exécution temps réel** → ajouté `calc_on_every_tick=true`
4. **`array.push` avec virgules** → erreur compilation silencieuse → séparés ligne par ligne
5. **Fonctions mutuellement récursives** (`lab_update` appelait `lab_reset`) → Pine Script v5 interdit → logique inline
6. **`var` déclarés après blocs `if`** → erreur parser → toutes les `var` déplacées en tête du script
7. **`array.new_float(4, 1.0)` (2 arguments)** → cause "Syntax error at input 'nclosed'" → remplacé par `array.new_float(0)` + init via `barstate.isfirst`
8. **`strategy.initial_capital` dans `var` init** → remplacé par `0.0` + mise à jour sur `barstate.isfirst`

### Structure finale du script (commit 7325bdf)
- Toutes `var` en haut, sans type explicite
- `array.new_float(0)` + `barstate.isfirst` pour init séquence [1,1,1,1]
- `bet_cur` calculé via deux `if` séparés (pas de ternaire imbriqué)
- `if` séparés (pas `if/else`) dans le bloc Labouchere update
- `calc_on_every_tick=true` + `process_orders_on_close=true`

### Leçon apprise — Pine Script v5
- `array.new_float(size, init_value)` à 2 arguments peut planter le parser silencieusement
- Toujours utiliser `array.new_float(0)` et init via `barstate.isfirst`
- Les `var` DOIVENT être déclarées avant tout code exécutable (blocs `if`, assignments)
- Fonctions utilisateur ne peuvent pas s'appeler mutuellement si elles modifient des variables globales
- Éviter les `if/else` imbriqués → préférer deux `if` séparés pour éviter les problèmes de type retour
- `var int x = 0` → préférer `var x = 0` (sans type explicite)
